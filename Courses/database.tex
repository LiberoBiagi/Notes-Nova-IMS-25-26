\section{Database}


\subsection{Introduction}

\subsubsection{What is a database}

A database can be defined as a collection of related data items within a specific business.
DBMS are sotwares that define, create, use and maintain a database, if we combine a DBMS with a database we get a database system
The majority of DBMS softwares rely on SQL systems. 

\subsubsection{Conceptual Modelling of a database}

\vspace{10pt}

To work with databases we need to model them to understand how they work. Modelling means representing reality in a 
scaled version. The same with databases we connect information to represent how relationships in the world exist.
Based on the business needings we design the database in a logical fashion and we translate it to 
a our language of choice. The step is passing from conceptual design to logical design. The first is DBMS independent while the second is DBMS dependent.
The process is:

\begin{enumerate}
    \item Requirement collection and analysis
    \item Conceptual design
    \item Logical design
    \item Physical design
\end{enumerate}

\vspace{10pt}

An SQL database is based on entities and relationships. Entities are business concepts
with an unambiguous meaning to a particular set of users. Entities have certain characteristics called attributes.
The rows are the entities (also called instances). The columns are the attributes. An association 
between one or more entities is called relationship, it defines how tables are related to each other. 

\vspace{10pt}

Another useful concept is the one of cardinality, how many instances of the entities can be associated with each other. 

We can have
 \begin{itemize}
    \item 1 to 1
    \item 1 to many
    \item Many to many
    \item None to many
    \item 1 to None
\end{itemize}

\vspace{10pt}

\subsubsection{Normalization}

An important step in the design is the one related to data normalization, we want to avoid redundant data. We split 
the tables into smaller tables to avoid anomalies during insertion, deletion or updates. The 
procedure follows a step by step transformation. The database is more complex from the structural point of view but can be easier to manage.

We have 3 possible normal forms.

\begin{itemize}
    \item First NF \ra Each cell in the table can have only one value, never a list of values 
    \item Second NF \ra An entity type is in 2NF when it is in 1NF and when all of its non-key
attributes are functionally dependent on its primary key. Features should be fully dependent on the entire primary key
    \item Third NF \ra An entity type is in 3NF when it is in 2NF, and non-key attributes are
directly (non-transitively) dependent on the primary key.
In other words: non-prime attributes must be functionally dependent on the
key(s), but they must not depend on another non-prime attribute.

\end{itemize}

\subsection{Database modelling}

\subsubsection{Architecture}


All DBMS follow a certain structure made by certain elements

\includegraphics[width=16cm]{dbms structure.png}


\begin{itemize}
    \item \textbf{Connection manager} \ra provides facilities to set-up a database connection. It can be
set-up locally or through a network
    \item \textbf{Security manager}\ra verifies whether a user has
the right privileges to execute the database actions required
    \item \textbf{DDL compiler}\ra compiles the data definitions specified in
DDL. Most relational databases use SQL as their DDL
    \item \textbf{Utilities}\ra A loading utility (load data from a variety of sources),
reorganization utility (reorganizes the data), user management utilities (support the
creation of user groups or accounts), etc
    \item \textbf{DML compiler}\ra compiler compiles the data manipulation
statements specified in DML
    \item \textbf{Query parser}\ra parses the query into an internal representation format that can
then be further evaluated by the system. It checks the query for syntactical and
semantical correctness
    \item \textbf{Query rewriter}\ra  optimizes the query, independently of the current database state. It
simplifies it using a set of predefined rules and heuristics that are DBMS-specific
    \item \textbf{Query optimizer}\ra optimizes the query based upon the current database state. It can
make use of predefined indexes that are part of the internal data model and provide
quick access to the data
    \item \textbf{Query executor}\ra takes care of the actual execution by calling on the storage
manager to retrieve the data requested
    \item \textbf{Transaction manager}\ra supervises the execution of database transactions.
Remember, a database transaction is a sequence of read/write operations considered
to be an atomic unit
    \item \textbf{Buffer manager}\ra is responsible for managing the buffer memory of the DBMS. The
DBMS checks first the memory when data need to be retrieved. Retrieving data from the
buffer is significantly faster than retrieving them from external disk-based storage
    \item \textbf{Lock manager}\ra essential component for providing concurrency control, which
ensures data integrity at all times
    \item \textbf{Recovery manager}\ra supervises the correct execution of database transactions. It keeps
track of all database operations in a logfile and will be called upon to undo actions of
aborted transactions or during crash recovery
\end{itemize}

A DBMS needs to interact with various parties, such as a database designer, a
database administrator, an application, or even an end-user (Ex. APIs).

\subsubsection{CRUD operations}


CRUD operations refere to those operations that are meant to Create, Read, Update and Delete data in our database.

\vspace{10pt}

To insert data

\begin{verbatim}
    INSERT INTO table_name(c1, c2,...)
    VALUES (v1,v2,...);
\end{verbatim}

To read data

\begin{verbatim}
    SELECT column_1, column_2,\dots
    FROM table_1
    [INNER|LEFT|RIGHT] JOIN table_2 ON condition
    WHERE conditions
    GROUP BY column_1
    HAVING group_conditions
    ORDER BY column_1
    LIMIT offset, length;
\end{verbatim}

To update our data

\begin{verbatim}
    UPDATE [LOW_PRIORITY] [IGNORE] table_name
    SET column_name1 = expr1,
    column_name2 = expr2,
    [WHERE condition];
\end{verbatim}

To delete data

\begin{verbatim}
    DELETE FROM table_name
    WHERE condition;
\end{verbatim}


\subsubsection{SQL clauses and operators}

The main clauses and operators that we have in SQL are

\begin{itemize}
    \item LIMIT
    \item DISTINCT
    \item ORDER BY
    \item BETWEEN
    \item [NOT] LIKE
\end{itemize}

\vspace{10pt}

LIMIT is used in the select statement to costrain the number of rows to return

\begin{verbatim}
    SELECT select_list
    FROM table\_name
    LIMIT [offset] row_count;
\end{verbatim}

Offset specifies the offset of the first row to return (from 0) while row\_count the maximum number of rows to return

\vspace{10pt}

LIMIT will remove duplicates

\begin{verbatim}
    SELECT DISTINCT e.first_name FROM employees AS e LIMIT 10;
\end{verbatim}

\vspace{10pt}

The BETWEEN operator is a logical one that we can use to specify a value within or not a range

\begin{verbatim}
SELECT e.emp_no, e.first_name, e.last_name, e.hire_date
FROM employees AS e
WHERE e.hire_date BETWEEN '1999-01-01' AND '1999-12-31';
\end{verbatim}

\vspace{10pt}

The LIKE operator is a logical operator that tests whether a string
contains a specified pattern or not

\begin{verbatim}
SELECT e.first_name, e.last_name
FROM employees AS e
WHERE e.last_name LIKE 'T%';


SELECT e.first_name, e.last_name
FROM employees AS e
WHERE e.last_name NOT LIKE 'Tr%';
\end{verbatim}

We can use the keyword AS to refere in a different way

\begin{verbatim}
table_name AS table_alias

SELECT
[column_1 | expression] AS descriptive_name
FROM table_name;

SELECT
e.firstName, e.lastName
FROM employees e
ORDER BY e.firstName;

SELECT
CONCAT (lastName, ' ', firstname) AS Full_name
FROM employees;

\end{verbatim}

The IN operator allows you to determine if a specified value matches
any value in a set of values or returned by a subquery.

\begin{verbatim}
SELECT column1,column2,...
FROM table_name
WHERE column_n IN (value1,value2,...);
\end{verbatim}

The SELECT statement does not provide the result set sorted.
To sort the result set, you add the ORDER BY clause to the SELECT statement

\begin{verbatim}
SELECT select_list
FROM table_name
ORDER BY column1 [ASC|DESC], column2 [ASC|DESC], ...;

\end{verbatim}

\subsubsection{SQL joins}

There are four basic types of SQL joins: inner, left, right, and
full.


\vspace{10pt}

INNER
\begin{verbatim}
SELECT column_list
FROM table_A
INNER JOIN table_B ON join_condition;

SELECT column_list
FROM table_A
INNER JOIN table_B USING (column_name);

SELECT column_list
FROM table_A A, table_B B
WHERE A.column_name = B.column_name;

SELECT column_list
FROM Table_A A
INNER JOIN Table_B B
ON A.Key = B.Key;
\end{verbatim}

LEFT

\begin{verbatim}
SELECT column_list
FROM Table_A A
LEFT JOIN Table_B B
ON A.Key = B.Key;
\end{verbatim}

RIGHT

\begin{verbatim}
ELECT column_list
FROM Table_A A
RIGHT JOIN Table_B B
ON A.Key = B.Key;
\end{verbatim}

OUTER/FULL

\begin{verbatim}
SELECT * FROM t1
LEFT JOIN t2 ON t1.id = t2.id
UNION ALL
SELECT * FROM t1
RIGHT JOIN t2 ON t1.id = t2.id
\end{verbatim}

Left excluding

\begin{verbatim}
SELECT column_list
FROM Table_A A
LEFT JOIN Table_B B
ON A.Key = B.Key
WHERE B.Key IS NULL;
\end{verbatim}

Right excluding

\begin{verbatim}
SELECT column_list
FROM Table_A A
RIGHT JOIN Table_B B
ON A.Key = B.Key
WHERE A.Key IS NULL;
\end{verbatim}

Self 

\begin{verbatim}
SELECT m.lastName AS Manager, e.lastName AS 'Direct report'
FROM employees e
INNER JOIN employees m ON m.employeeNumber = e.reportsTo;
\end{verbatim}

\subsubsection{Functions}

Functions in SQL can be

\begin{itemize}
    \item Aggregate \ra Performs a calculation on multiple values and returns a single value. Can be AVG(), MAX(), MIN(), STDEV(), COUNT() or SUM(). COUNT() has three main forms
    \begin{enumerate}
        \item COUNT(*) \ra returns the number of rows including duplicates, non-NULL and NULL rows.
        \item COUNT(expression) \ra returns the number of rows with non NULL values
        \item COUNT(DISTINCT expression) \ra returns the number of distinct rows that do not contain NULL values
    \end{enumerate}
    \begin{verbatim}
        SELECT count(employee_id) as employees
        FROM employee
        WHERE department_id = 50;
    \end{verbatim}

    \item Math \ra mathematical computations. Can be MOD(), ROUND(), TRUNCATE(), SIN(n), SQRT(n), RAND(), LOG(n) or POW()
    \begin{verbatim}
        SELECT round(salary, 2)
        FROM employee;
    \end{verbatim} 

    \item Date \ra make computations on date values. Can be CURDATE()/CURRENT\_DATE(), DATEDIFF(), DAY(), DAYOFWEEK(), NOW(), MONTH(), TIMEDIFF() or YEAR().
    
    \item String \ra work on strings. Can be CONCAT(), LENGTH(), LOWER(), REPLACE(), SUBSTRING(), TRIM(), UPPER() or INSTR()

\end{itemize}

\subsubsection{Group By}

The GROUP BY clause tries to take a set of rows and summarize them by values. It returns one row for each group. Is used with SUM, AVG, MAX, MIN and COUNT.

\begin{verbatim}
    SELECT c1, c2, ..., cn, aggregate_function(ci)
    FROM table_name
    WHERE where_conditions
    GROUP BY c1, c2, ..., cn;
\end{verbatim}

\subsubsection{Having}

The HAVING clause is used in the SELECT statement to specify filter conditions for a group of rows or aggregates. Oten used with GROUP BY to filter groups based on a specific condition. HAVING applies a filter condition to each group of rows, while the WHERE does that to each individual row.

\begin{verbatim}
    SELECT select_list
    FROM table_name
    WHERE search_condition
    GROUP BY group_by_expression
    HAVING group_condition
\end{verbatim}

\subsubsection{Views}

A VIEW is defined by means of an SQL query and its content is generated upon invocation of the view by an application or other query. Queries can be run against the view. A view will not physically store the data. Each time a view is queried the underlying query called view definition is executed. We create views to do queries that we might want to do frequently.

\begin{verbatim}
    CREATE[OR REPLACE] VIEW [db_name.]view_name[(column_list)]
    AS select-statement
\end{verbatim}

\subsubsection{Triggers}

A TRIGGER is a piece of SQL code made of declarative and/or procedural instructions and stored in the catalog of the RDBMS. Is automatically runned whenever a specific event (INSERT, UPDATE, DELETE) occurs and a specific condition is evaluated as true.

\begin{verbatim}
    CREATE TRIGGER trigger_name trigger_time trigger_event
    ON table_name
    FOR EACH ROW
    BEGIN what trigger does
    END;
\end{verbatim}

Where trigger time can be BEFORE or AFTER. 

Where trigger event can be INSET, UPDATE or DELETE.

\vspace{10pt}

Advantages

\begin{itemize}
    \item Aumatization and verification in case of specific events or situations
    \item Can model more rules without change the code
    \item Can assign default values for new tuples
    \item Synchronic updates in case of data replication
    \item Automatic auditing and logging
    \item Automatic exporting of data
\end{itemize}

Disadvantages

\begin{itemize}
    \item Hidden, difficult to manage
    \item Cascade effects
    \item Uncertain outcomes 
    \item Deadlock situations
    \item Debugging complexities
    \item Maintainability problems
\end{itemize}

\subsection{Query Optimization}


\subsubsection{The problem}

Users can complain that applications are slow. One reason can be that the queries in the SQL database are slow and not optimized.

User's feeling about response time
\begin{itemize}
    \item 0.1 second \ra feels lika instantaneous
    \item 1 second \ra limit to that feeling
    \item 10 seconds \ra the user will do something else
\end{itemize}

A query of 3.19 seconds can be considered as slow, we need to understand if it has problems. We can do that by using a QEP (Query Execution Plan). A query plan is a list of instructions that the database needs in order to execute a query on the data. The query optimizer will generate multiple query plans for a single query and will select the most efficient one.

\begin{verbatim}
    SELECT *
    FROM tools
    WHERE name="Screwdriver";
\end{verbatim}

vs

\begin{verbatim}
    SELECT * 
    FROM tools 
    WHERE id = 3;
\end{verbatim}

Both are returning the same results but the second one because the index is usually unique and comparisons between strings is slower. 

\subsubsection{EXPLAIN}

The EXPLAIN command is used to show query plans. It doesn't run the actual SQL statement (except subqueries). It doesn't provide any tuning recommendation but information that can be used to do that. For example we can use a newly created index with an alter statement. 

How to identify problems using EXPLAIN
\begin{itemize}
    \item No index used (NULL in the key column)
    \item A lot of rows are processed (rows column)
    \item A lot of indexes are evaluated (possible\_keys column)
\end{itemize}

Remember

\begin{itemize}
    \item We can optimize queries in many ways
    \item The generated query plan can change depending o many factors
    \item For UPDATE and DELETE statements we need to rewrite the query as a SELECT statement to confirm index usage
\end{itemize}

\subsubsection{Indexes}

We need to be sure that our indexes make sense. An index is a structure that holds the field the index is sorting and a pointer from each record to their corresponding record in the original table where the data is stored. The id is incrementing based on the order in which the data was added. To fasten the search the data are organized like a binary tree based on the middle entry.
Indexing makes columns faster to query by creating pointers to where data is stored within a database.

\vspace{10pt}

Index cardinality (amount of unique values)
\begin{itemize}
    \item Multiple different indexes make MYSQL try to identify the most effective index for the query.
    \item The optimizer chooses an index based on the estimated cost to do the least amount of work
    \item Index cardinality can be used to to that
\end{itemize}


Normally indexes are created when the table is created. After we can do that using

\begin{verbatim}
    CREATE INDEX index_name ON table_name
    (column_list)
\end{verbatim}

\subsubsection{Tips}

\begin{enumerate}
    \item Define SELECT fields instead of SELECT *
    \item Avoid SELECT DISTINCT if possible
    \item Use WHERE instead of HAVING
    \item Use WILDCARDS at the end of the phrase
    \item Use LIMIT
    \item Run queries during off-peak times
    \item Index the queries properly \ra the column is queried frequently, foreign keys to other tables and a unique key exists on the columns
\end{enumerate}
